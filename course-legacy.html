Session 1 — The Loop

By the end of this session, you will have built a working AI agent in ~40 lines of code and understood exactly why every line is there.

The Beginner Build Box

Before we write code, we need to understand what an AI agent actually does. Open the Workshop Console (or run node server.js in the ai-course folder and visit http://localhost:8000).

This is a visual "Build Box" that abstracts away the code. It teaches you the core workflow: 1. Goal: What you want to achieve. 2. System Rules: How the AI should behave. 3. Context / Notes: Information the AI needs to know to do the job. 4. Acceptance Checks: How you define "done".

Run a job in the Build Box and watch the terminal trace on the right. You'll see the AI plan, draft, check, and finalize. That workflow is the secret to building reliable AI tools. When you run a loop in Python, you're just doing exactly what the Build Box did automatically.

Cold open: the same question, five times

We ask a model the same question five times and watch the answers differ. This is not a bug and it never goes away. Every design decision in this course exists because the thing you're building on is non-deterministic.

If you take one habit from today: run it more than once before you believe it.

What a model actually is

Strip the mystique and you have a function:

text in -> probability distribution over the next token -> sample one -> repeat

Consequences that matter, in order of how often they bite people:

  • It has no memory. None. A "conversation" is your app resending the entire transcript every turn. When ChatGPT remembers your name, some code stored that and pasted it back in. You'll write that code today.
  • The context window is the whole world. If it's not in the prompt, it doesn't exist to the model — no matter how true or important it is.
  • It can't do anything. A model emits text. It cannot read your file or send your email unless you wrote a function that does that and wired it up.
  • Same input, different output. See above. Plan for it.

Tokens: models see chunks of characters, not words — roughly 4 characters per token in English. Useful because you pay per token, latency scales with tokens, and the context window is measured in them. It also explains the classic party-trick failures: counting letters in a word is hard when you can't see letters.

The main event: recursive prompting vs a loop

Recursive prompting — you are the loop

  1. Ask the model something
  2. Read the answer
  3. Copy it into a terminal or a doc
  4. Copy the result back into the chat
  5. "That's not quite right, try again"
  6. Repeat until it's good or you give up

An agent loop — code is the loop

  1. Give the model a goal and some tools
  2. It requests a tool call
  3. Your code runs it
  4. Your code feeds the result back
  5. Repeat until it stops asking
  6. Go do something else meanwhile

The difference isn't convenience. Three things change qualitatively:

  • It can react to what it finds. With copy-paste you decide the next step. In a loop the model sees the actual output — the real error, the real file contents — and adapts. You never wrote that branching logic.
  • It runs while you don't. Twenty steps at 3am is the same effort as one.
  • It's a program. Programs can be versioned, tested, scheduled, and handed to a colleague. Your clipboard technique can't.

The loop, in full:

while True:
    response = model(messages, tools)
    messages.append(response)

    if response.stop_reason != "tool_use":
        break                      # model has nothing left to ask for

    for call in response.tool_calls:
        result = run(call)         # YOUR code, YOUR permissions
        messages.append(result)    # goes back in as a user turn

That is the architecture behind every coding agent, research agent, and "autonomous" product you've read about. The rest is tools, prompts, and guardrails.

Break What you should learn

Session 2 — Context & Memory

By the end of this session, you will know how to give an AI access to your own private documents, and how to build a memory system that remembers facts across sessions.

The model only knows what's in front of it. Your job is deciding what goes there.

Cold open: your homework, debugged live

Two volunteers. We look at what actually went into the context window when it failed. Nine times out of ten the fix is there and not in the prompt wording.

Context engineering

"Prompt engineering" undersells it. In a real system the prompt is assembled at runtime from many sources, and the assembly is the engineering:

[ system prompt      ]  who it is, what it must never do
[ tool definitions   ]  what it can do
[ retrieved chunks   ]  facts pulled for THIS question      <- RAG
[ memory             ]  durable facts about this user
[ conversation       ]  what's been said so far
[ the actual question]

Two constraints shape everything:

  • Space. The window is finite, and cost and latency both scale with what you put in it.
  • Attention. More context is not better. Bury the important line among forty irrelevant ones and the model will miss it — the same way you would. The degradation on long, cluttered contexts is often called context rot. Precision beats volume.

The whole game: get the right few thousand tokens into the window, and as little else as possible.

RAG, demystified

Retrieval-Augmented Generation is one sentence: search for relevant text, paste it into the prompt, ask the question. That's the entire idea. Intimidating name, about fifteen lines of core logic.

question -> search your documents -> top 5 chunks -> stuff into prompt -> answer

The interesting part: search is where all the difficulty lives, and search is a well-understood forty-year-old discipline with nothing to do with AI.

Chunking — the decision nobody talks about. You can't retrieve whole documents, so you split them. How you split determines your ceiling; no embedding model rescues bad chunks.

  • Too small: precise matches, but the retrieved piece lacks the context to be useful
  • Too big: full of context, but one relevant sentence gets diluted by nine irrelevant ones
  • Best default: split on structure a human already put there — headings, sections, slides. In a notes vault a ## heading is a free semantic boundary. Use it.

Two ways to search, and neither wins:

Keyword (BM25) Embeddings
finds exact words, rare terms similar meaning
great at names, IDs, error codes, jargon vague or fuzzy questions
fails at synonyms — "salary" misses "compensation" exact strings — invoice #4471
cost free, instant, no deps needs a model, an index, maintenance

Most teams jump straight to a vector database and never try keyword search. For a few thousand documents BM25 is often as good, and it's twenty lines with no infrastructure. Try the boring thing first. Production systems typically run both and merge.

Memory

RAG answers "what do my documents say?". Memory answers "what do you know about me?". Different problem: memory is small, written by the assistant, and must survive across sessions.

There are two main patterns for giving an AI agent persistent memory: using raw markdown files, and using an integrated service like GBrain.

Pattern 1: Obsidian (Files and Folders)

For personal and small-team memory, markdown files in a folder beat a database on nearly every axis that matters: - You can read them. Try inspecting what a memory product decided to store about you. - You can edit them when they're wrong — with a text editor, right now. - git diff shows exactly what your assistant learned this week. - They're yours. No lock-in, readable in 2040.

Obsidian is a local-first markdown knowledge base that provides a lovely front end for these files. - What it is: A free tool to read, write, and link markdown files. - Install: Download from obsidian.md or run brew install --cask obsidian on Mac. - How to use it: Create a "vault" (which is just a folder on your computer). Then point our Python agent at that folder using export MEMORY_DIR=~/path/to/your/vault. The agent will read and write .md files right into your vault.

Pattern 2: GBrain (Managed Memory)

If you are building in the Google ecosystem, GBrain (Gemini's long-term context memory system) offers a managed alternative. - What it is: Instead of reading and writing raw .md files, the Gemini model inherently remembers context, preferences, and facts about you across sessions. - How to use it: When using Gemini, you can often reference your "brain" or @gemini to pull in long-term context. - The Tradeoff: GBrain is effortless and scales well, but it is less manually inspectable than opening a markdown file.

Which should you use? Start with files (Obsidian) so you can physically see how memory works. Graduate to managed services (GBrain) when managing the files becomes tedious.

Three rules that make memory work

  • Write self-contained sentences. "Caleb decided on usage-based pricing", never "he decided on that". Memory is read out of context by definition.
  • Date everything. Undated memory rots invisibly — you can't tell a current fact from a superseded one.
  • Retrieval discipline beats storage. Memory that never gets looked up is a diary. The system prompt telling it when to search matters more than the tools.

Session 3 — Shipping Internal Tools

By the end of this session, you will understand the difference between a flaky demo and a reliable tool, and you will build a complete workflow for your own job.

The distance between a demo that works and a tool people trust.

Cold open: a week of memory

We look at someone's real memory folder after a week. Duplicates, stale facts, junk topics. Nothing was broken — this is what unmaintained AI systems look like, and it's why the rest of today is about the unglamorous parts.

The gap: a demo needs to work once, for you, with you watching. An internal tool needs to work most times, for people who won't notice when it's wrong. That second requirement is a completely different engineering problem.

Evals

An eval is a list of inputs and what a correct output looks like. That's it. You don't need a framework; you need to write down what "correct" means, which is genuinely hard and is precisely why people skip it.

Without evals you're doing this: change the prompt, try one example, it looks better, ship it. You have no idea if you improved anything or broke three other cases.

What makes a suite good:

  • Hard cases, not easy ones. Ten nasty examples beat a hundred obvious ones. Ambiguous inputs, multi-issue inputs, the polite email that's actually an emergency.
  • Run each case several times. Non-determinism means a case can pass at 60%. A suite that runs each case once will lie to you with a straight face.
  • Write them before you tune, or you'll unconsciously write tests your current prompt already passes.
  • Anything under 100% is a defect, not noise.

Guardrails

Assume the model will occasionally do the wrong thing, because it will. Design so that's survivable.

Principle In practice
least privilege Read-only DB user. Scoped API token. A folder, not a filesystem.
reversible Draft, don't send. Stage, don't commit. Soft delete.
human gate Anything that spends money, emails a customer, or deletes data.
constrain in code If the tool can't do it, no prompt injection can make it happen.
log everything Full transcripts. When it misbehaves you need to see what it saw.
budget the loop Turn limits, token limits, timeouts. Runaway loops are expensive.

Prompt injection: if your agent reads untrusted text — web pages, incoming email, user uploads — assume that text can contain instructions and that the model may follow them. There's no reliable prompt-level fix. The mitigation is architectural: limit what the tools can do, and gate anything irreversible behind a human.

MCP, briefly

You wrote tools by hand in session 1. MCP (Model Context Protocol) is a standard way to package tools so any AI client can use them, rather than each app reimplementing a Slack integration. It's a plug format, not magic.

Practical advice: use existing MCP servers, write your own only when nothing fits. Most of what you want already exists. And the same guardrails apply — a server someone else wrote still runs with whatever permissions you hand it.

Choosing what to build (3 Worked Examples)

The best internal tools are boring and specific. Good candidates take text in, apply a set of rules, and output text that a human reviews. Not everything should be an AI feature — knowing when NOT to use a model makes you credible.

Here are three concrete examples of what a reliable tool looks like. Notice the structure: Input -> Rules -> Checks -> Output.

Example 1: Meeting Follow-Up Extractor

  • Goal: Turn messy raw notes into a clean list of action items.
  • Input: Raw stream-of-consciousness meeting notes.
  • System Rules: "You are a practical assistant. Separate decisions from tasks. Keep it short."
  • Acceptance Checks (Evals):
  • Does every action have an owner or explicitly say 'owner needed'?
  • Are decisions clearly separated from simple tasks?
  • Is it short enough to paste directly into Slack?
  • Output: A formatted bulleted list ready for review.

Example 2: Customer Email Triage

  • Goal: Classify incoming emails to route them correctly.
  • Input: "We were charged twice for invoice 4471 and the export is missing data. Finance is blocked."
  • System Rules: "Reply with JSON only containing category, urgency, and a one-sentence summary. Urgency is high only if a customer is blocked or money moved."
  • Acceptance Checks (Evals):
  • Is the output strictly valid JSON?
  • Does the urgency correctly identify the billing/blocking issue?
  • Output: {"category": "billing", "urgency": "high", "summary": "Customer charged twice for invoice 4471 and export is broken."}

Example 3: Invoice Explainer

  • Goal: Help non-technical staff understand complex cloud billing line items.
  • Input: A dense JSON or CSV dump of AWS/GCP line items.
  • System Rules: "Explain these charges to a small business owner. Do not use acronyms without defining them."
  • Acceptance Checks (Evals):
  • Are all acronyms defined?
  • Is the total cost accurate and easy to find?
  • Is the tone helpful and non-technical?
  • Output: A three-paragraph plain-English summary of where their money went.

Glossary

A quick reference for the terms used throughout this course.

  • Token: The fundamental unit a model reads and writes. Roughly 4 characters or 3/4 of a word in English. Models do not see words or letters.
  • Context Window: The model's short-term memory. If it is not in the context window, the model does not know it. Measured in tokens.
  • RAG (Retrieval-Augmented Generation): A pattern for giving models facts. You search a database, copy the relevant chunks of text, paste them into the prompt, and ask the model to read them.
  • Chunking: The process of breaking a large document into smaller pieces (chunks) so they can fit into the context window when retrieved via RAG.
  • BM25: A classic keyword search algorithm. Excellent for finding exact names, IDs, and error codes.
  • Embedding / Vector Search: A way of turning text into a list of numbers so you can search by meaning rather than exact words. Excellent for vague questions, bad at exact IDs.
  • Prompt Injection: When untrusted text (like an incoming email) contains instructions that the model follows instead of your system prompt.
  • MCP (Model Context Protocol): An open standard for packaging tools so that any AI assistant can use them, removing the need to write custom tools from scratch.

Cheat sheet

Mental models

  • A model is a function from text to text. It has no memory and can't do anything on its own.
  • An agent is a while loop around that function with tools attached. Forty lines.
  • The context window is the model's entire world. If it's not in there, it doesn't exist.
  • RAG is search plus paste. The hard part is the search.
  • Memory is a folder of markdown files until you prove otherwise.
  • An eval is a list of cases and expected outputs. No framework required.

Rules of thumb
run it 3× Non-determinism means one run tells you almost nothing.
tool > prompt Never enforce in the prompt what you can enforce in code.
errors in-band Return errors to the model as text so it can recover.
boring search first Try BM25 before you stand up a vector database.
less context Precision beats volume. Long cluttered prompts degrade.
date your memory Undated facts rot without telling you.
self-contained facts "Caleb chose X", never "he chose that".
evals before tuning Or you'll write tests your prompt already passes.
gate the irreversible Money, customers, deletion — human in the loop.
log transcripts You can't debug what you can't see.

Debugging checklist — when an AI system misbehaves, in this order:

  1. Print the full context. What actually went into the window? Usually the answer is here.
  2. Check retrieval. Was the needed information even retrieved? Inspect the chunks.
  3. Run it 3 more times. Consistent failure or flaky? Completely different bugs.
  4. Read the tool descriptions as if you'd never seen them. Ambiguous?
  5. Check the tool results. Did a tool silently return an error string the model then trusted?
  6. Only now edit the prompt — and add an eval case first, so you can tell if it helped.
